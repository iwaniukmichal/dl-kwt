# Experiment Report Runner

## Goal

Use an already implemented repository and README to:

1. identify the experiments the project requires,
2. inspect the existing result files in `outputs/`,
3. collect the metrics/artifacts produced by the user,
4. and write the final project report into `docs/report/report.tex`.

The report must present **insight, not raw data**. Every table, figure reference, and written paragraph should communicate a conclusion, comparison, or interpretation grounded in the discovered artifacts. Moreover every experiment that has been conducted must be documented in the report. 

The report should be treated as a **complete research study**, not as a template to be mechanically filled with the user's produced results. In addition to reporting empirical outputs, it must include the necessary background, methodological framing, theoretical motivation, and discussion expected in a serious academic report. Filling the report may therefore require additional focused research on the relevant methods, publications, baselines, and theoretical foundations.

## Required inputs

The repository should contain or provide:

- the project plan file provided by user,
- the report outline/template in `docs/report/report.tex` — this defines the required LaTeX sections,
- a `README.md` describing the expected experiments and workflow,
- an `outputs/` directory containing experiment artifacts generated by the user,
- the implemented codebase

## Source-of-truth priority

Use sources in this order:

1. explicit user instructions,
2. the **project plan** (`plan.tex`) — defines what must be reported, the structure, the phases, and the research questions,
3. the **report outline/template** (`docs/report/report.tex`) — defines the required LaTeX sections,
4. the README experiment instructions — defines the expected experiments and workflow,
5. the actual files inside `outputs/` — the sole evidence for every quantitative claim,
6. repository code/config — for understanding provenance and experiment parameters.

For **what should be reported and how it should be structured**, use the project plan and the report template.
For **what actually happened**, use only the actual artifacts found in `outputs/` and computations derived from those artifacts.

## Core rules

1. **Report what exists, not what could be rerun.**
   Do not run experiments by default. Use only the artifacts already present in `outputs/` unless the user explicitly asks for execution.

2. **Use `plan.tex` as the structural guide for content.**
   The project plan defines the phases, experiments, research questions, and analyses. The report content must follow the plan's logical structure. The plan's research questions must be addressed.

3. **Use the report template for LaTeX section layout.**
   The report template (`docs/report/report.tex`) defines the required top-level LaTeX sections. Fill each section with content derived from the plan's structure and the discovered outputs.

4. **Treat `outputs/` as the evidence base.**
   Every result, metric, figure, and claim in the report must come from actual files in `outputs/` or from computations derived from them.

5. **Derive meaningful computations.**
   Go beyond what is stored in individual output files. Required derived computations include (when the underlying data exists): comparisons, rankings, best configurations, etc.

6. **Do not invent missing results.**
   If an expected experiment has no matching outputs, explicitly state it is missing. Do not fabricate metrics or interpretations.

7. **Prefer minimal repository intervention.**
   Do not redesign the repository. Only create small helper scripts if strictly necessary to aggregate or parse existing output files.

8. **Use `latex-writing-skill` for LaTeX quality.**
    When writing or revising `docs/report/report.tex`, follow the conventions and style rules from `latex-writing-skill`.

9. **Treat the report as a full research document.**
   Do not limit the report to inserting experiment outputs into empty sections. The final document should contain the explanatory and analytical material expected in an academic study, including problem context, methodological rationale, theoretical background, discussion of related approaches, and interpretation of results.

10. **Perform additional research when needed to complete the study.**
   If the report requires method descriptions, theoretical explanations, literature context, or publication-backed framing that is not already present in the repository, perform focused additional research and incorporate it into the report. Use such research to support background and interpretation sections, while keeping all quantitative experimental claims grounded strictly in the discovered artifacts.

 11. **All experiments must be explicitly reported.**
   Every experiment that has been conducted must be documented in the report. This includes a clear description of the experimental setup and experiment-specific configurations and hyperparameters in a concise and structured form, preferably as a table summarizing key settings and/or text description. The report therefore serves as the single source of truth for all experimental work.
